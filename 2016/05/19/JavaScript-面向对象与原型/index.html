创建对象

创建一个对象,给这个对象新建属性和方法

基本模式

不难发现,普通模式创建对象是有一些弊端的,如:

  • 多个类似对象的声明是很复杂的
  • 没有面向对象的概念,对象没有被封装起来

工厂模式

针对一般创建函数的方式的一些弊端,我们可以使用一种叫做工厂模式的方式来解决实例化对象大量重复的问题

虽然工厂模式解决了重复实例化的问题,但是有一个明显的问题就是无法识别新建的实例到底是哪个对象的实例.

构造函数模式

EAMAScript中还可以使用构造函数来创建特定的对象(类似Object对象)

使用构造函数的方法和工厂模式有明显的不同,在于:

  • 构造函数没有显式的创建对象(new Object())
  • 直接将属性和方法赋值给this对象
  • 没有return语句

构造函数方法的一些规范

  • 函数名和实例化构造名必须大写 有利于区分
  • 通过构造函数创建对象，必须使用new运算符

关于 this 的使用

  • 在全局范围时this就代表window对象
  • 在构造函数体内就代表构造函数所声明的对象

构造函数与普通函数的唯一区别就是构造函数必须 new 出来

原型

我们创建的每一个函数都有一个 prototype 属性 这个属性是一个对象，用途是可以包含所有实例共享的属性和方法。使用原型的好处在于可以让所有实例对象共享属性方法，无需一个一个添加
在构造函数和工厂模式中,我们每new一个实例,就会在内存中开辟一个新的区域用来存放属性与方法.
实际上有些属性方法是共有的,这时候我们可以引入prototype了

原型的组成

由上诉例子,我们发现两个实例的方法指向同一块区域

基本的创建方法

但实际上,通过字面量的形式创建对象原型时,对象原型的constructor会指向Object,而非构造函数本身

我们也可以通过直接强制指向构造函数对象本身

PS：字面量方式为什么constructor会指向Object？因为Box.prototype={};这种写法其实
就是创建了一个新对象。而每创建一个函数，就会同时创建它prototype，这个对象也会自
动获取constructor属性。 所以， 新对象的constructor重写了Box原来的constructor， 因此会
指向新对象，那个新对象没有指定构造函数，那么就默认为Object。
当我们在重写原型时,会切断原来的原型,不保留之前的属性方法

原型的优缺点

原型也有缺点,它省略了构造函数传参初始化这个步骤例:(this.age = age)，导致了初始化的值都是一样的,而这也是他最大的优点,那就是共享

还有一个缺点,原型中的属性如果包含 引用类型 ,就存在一定的问题了

组合构造函数+原型模式

针对构造函数传参和共享的问题,可以使用组合构造函数+原型模式

动态原型模式

原型模式中,无论是否调用了原型中的方法,他都会初始化方法,而且构造函数+原型部分有些奇怪,最好就是把构造函数和原型封装在一起,这时候动态原型模式就起作用了

通过以上方式,我们做到了封装,共享方法的一次声明并共享

寄生构造函数模式

寄生构造函数就是 工厂模式 + 构造函数模式，除非以上方法均无法实现需求，否则不要使用
那么，这种模式在什么情况下使用的多呢？
假设要创建一个额外方法的引用类型。由于不建议直接使用String.protoype.addString（因为不利于维护，还可能与原封装函数造成冲突），这个时候可以使用寄生模式

稳妥构造函数模式

在一些安全环境中，比如禁止 this 和 new 的

  • this 指的是在构造函数内部不使用this
  • new 指的是在外部实例化不适用new

借用构造函数(对象冒充)

为了解决 引用共享 和 子类无法给父类传参 的问题，我们可以采用一种叫 借用构造函数 的技术，或者称为 对象冒充 的技术来解决

组合继承(原型链 + 借用构造函数)

借用构造函数虽然解决了刚才两种问题，但是没有原型，也无法复用。这时候 组合继承 模式登场了

原型式继承

这种继承借助原型，同时基于已有的对象创建新的对象，同时还不必自定义类型

寄生式继承

寄生式继承 就是把 原型式继承 + 工厂模式,目的是为了封装创建对象的过程
